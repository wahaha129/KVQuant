## Section II: Technical Stress Test & Scalability（60%）

### Part A（系统重构）

用流程图表达（建议你自己画图）：
1. 离线阶段：`gradients -> quant`（Fisher + KMeans）
2. 在线阶段：`deployment` 每 token 做追加、稀疏结构更新、注意力计算
3. 标记关键路径：CPU-GPU 同步点、二阶段 kernel、LUT 反量化

可引用证据来源：
- `pipeline_integration_complexity.py`
- `sparse_overhead_simulation.py`

### Part B（假设与失效条件）

你可直接写 2–3 个“可攻击假设”：
1. **假设 A：在线追加开销可忽略**  
   - 反证：每 token 都可能触发 `outlier_count.to(kCPU)` 同步。
2. **假设 B：稀疏结构增长代价可忽略**  
   - 反证：`torch::cat` 对历史稀疏数据反复复制，累计开销上升。
3. **假设 C：4-bit 带宽收益可直接转化为吞吐收益**  
   - 反证：RoPE + LUT 反量化引入大量计算，未必仍是纯带宽受限。

可引用证据来源：
- `sparse_overhead_simulation.py`
- `dequantization_memory.py`

### Part C（可扩展性与敏感性）

你可以做“参数拉伸”讨论（无需大规模复现实验）：
- 上下文长度从 `1e4 -> 1e5 -> 1e6`，关注累计开销趋势。
- outlier rate 改变时，稀疏结构增长速度如何变化。
- 模型规模（7B/13B/65B）变化时，校准成本和部署复杂度如何变化。

可引用证据来源：
- `sparse_overhead_simulation.py`
- `pipeline_integration_complexity.py`

## 四、可直接套用的中文写作提纲（6–8页）

你可以按这个顺序写正文：

1. **引言（0.5页）**  
   - 研究问题：KV cache 量化在超长上下文是否“真实可扩展”？

2. **Section I 架构综合（1页）**  
   - 核心创新 + 效率前沿定位（容量/上下文长度 vs 延迟）。

3. **Section II-A 系统重构（1页）**  
   - 端到端流程 + 关键数据路径（离线/在线分离）。

4. **Section II-B 失效条件（1.5页）**  
   - 至少两个假设及其 breaking point（用脚本输出做证据）。

5. **Section II-C 可扩展性分析（1.5页）**  
   - 随 context/model size 变化的复杂度趋势讨论。

6. **Section III Weak Reject（约500词，约1页）**  
   - Hidden Costs / Methodological Gaps / Practicality 三段式。

7. **结论（0.5页）**  
   - 总结：方法在“容量扩展”有效，但“工程总成本与鲁棒性”存在显著脆弱点。

## 五、写作时的注意事项（避免丢分）

1. **不要写成论文复述**，要写“对抗性审计”。
2. **每个批评点都要绑定代码证据**（脚本输出 + 源码位置）。
3. **区分“猜测”与“已验证”**：  
   - 已验证：脚本统计和源码定位。  
   - 推断：你的临界点外推与规模化讨论（要注明假设）。
4. Defense 问答时，优先解释：  
   - 为什么显存收益不等于端到端收益；  
   - 为什么同步点和反量化会成为新瓶颈；  
   - 为什么工程复杂度可影响可部署性结论。
